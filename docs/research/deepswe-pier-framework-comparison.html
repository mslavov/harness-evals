Core finding: DeepSWE/Pier and harness-evals optimize for different evaluation shapes. DeepSWE/Pier is strongest for benchmark-style, single-task code correctness with independent hidden program verifiers. harness-evals is strongest for project-specific agent workflow regression, multi-step prompts, tool/mocking assertions, LLM judges, and native CLI comparisons.

What is a “test”? A Harbor-format task directory with instruction.md, task.toml, environment/, tests/, and held-out solution/. A YAML scenario/case, one or more prompt steps, optional fixture workspace, mocks, and assertions. DeepSWE tasks are portable benchmark units; our cases are project-local behavioral workflows.
Who decides correctness? An independent verifier script uploads hidden tests, applies a test patch, runs base and new tests, and writes a reward file. The runner evaluates step assertions over exit code, output, tool calls, mocks, workspace diff, and optional LLM judges. DeepSWE has stronger code-behavior verification out of the box; our framework has broader workflow observability.
How is score computed? Verifier reward values are aggregated by job metrics such as mean/min/max/sum and pass@k when binary rewards are available. Weighted score buckets: assertion pass rate, judge score, latency, cost, and token usage. DeepSWE is benchmark-leaderboard oriented; our scoring is multi-objective and tunable per project.
How are agents run? Pier runs trials through Docker or Modal environments, supports installed agents and mini-swe-agent, and records augmented ATIF trajectories. harness-evals runs selected adapters in Docker workspaces: command, Pi, Claude Code, Codex, Cursor, plus custom adapters. Both support real CLI agents. Pier has benchmark-scale trial orchestration; our harness has agent-first project setup and mocks.
What should we borrow? Independent verifier phase, reward files, pass@k attempts, model patch artifacts, flake checks, and prompt-verifier QA discipline. Keep multi-step scenarios, strict mocks, LLM judge assertions, weighted score buckets, current-auth mounting, and report exports. The best next step is adding a first-class post-agent verifier command/reward assertion to our framework.

DeepSWE + Pier mechanics

DeepSWE corpus and task setup

DeepSWE is a benchmark dataset of original long-horizon engineering tasks. The blog states that tasks span 91 active open-source repositories and five languages. The inspected repository contains 113 task directories with this language split:

35TypeScript
34Go
34Python
5JavaScript
5Rust

Every inspected task had allow_internet = false, agent.timeout_sec = 5400, verifier.timeout_sec = 1800, a prebuilt Docker image, a repository URL, and an immutable base commit.

Task artifactRole in the evalObserved DeepSWE pattern
task.tomlMetadata and runtime contract.Contains task id/title/category/language/repo/base commit; verifier, agent, and environment limits; Docker image and internet policy.
instruction.mdPrompt shown to the agent.Behavior-focused requirements. Internal helper names and implementation layout are usually left flexible.
environment/DockerfileReproducible image fallback.Clones upstream repo at the base commit and installs dependencies, e.g. go mod download.
tests/test.patchHidden verifier additions.Adds behavioral challenge tests plus a repository-local test.sh with base and new modes.
tests/test.shVerifier entrypoint.Captures model patch, resets test-touched files, applies test.patch, runs base/new tests, writes reward.
solution/Reference solution.Includes solution.patch and solve.sh; held out from grading and used for review/QA.

DeepSWE verifier sequence

  1. Start in the task repo. The verifier expects the app at /app in the sandbox.
  2. Capture the model patch. It soft-resets to the base commit, stages the model changes, and writes /logs/artifacts/model.patch.
  3. Reset hidden-test files. Files touched by /tests/test.patch are restored to the base state before applying tests.
  4. Apply the hidden test patch. git apply --whitespace=nowarn /tests/test.patch injects verifier tests and a repository-local test script.
  5. Run regression and challenge modes. The generated /app/test.sh base checks selected existing tests; /app/test.sh new runs the new behavioral tests.
  6. Write scalar reward. The task writes 1 to /logs/verifier/reward.txt only when both modes pass; otherwise it writes 0.
Quality model: DeepSWE’s blog emphasizes prompt-verifier alignment, acceptance breadth across valid implementations, realism, environment cleanliness, flake checks during authoring, LLM-assisted analysis, and human review. This QA discipline is as important as the runner itself.

Pier runner mechanics

Pier is the framework used to run Harbor-format tasks. It is a fork of Harbor with additions around installed CLI agents, air-gapped tasks, agent-specific network allowlists, augmented ATIF trajectory metadata, a trajectory viewer, and post-hoc critique jobs.

Pier componentMechanics
Job expansionJob resolves tasks from local datasets or explicit tasks, then expands n_attempts × tasks × agents. Concurrency defaults to four trials.
Trial lifecycleTrial starts environment, sets up agent, runs agent on the instruction, runs verifier, writes result.json, downloads logs/artifacts, and cleans up.
EnvironmentsDocker and Modal are supported. Task config controls image, build timeout, CPU/memory/storage, OS, internet, MCP servers, health checks, workdir, and optional verifier environment.
Network isolationWhen task internet is disabled, Pier can still provide filtered egress for installed agents via allowlisted provider/base URLs.
VerifierUploads tests/ to the environment, runs discovered test.sh or test.bat, downloads verifier output, and parses reward.txt or reward.json.
Multi-step tasksPier supports configured [[steps]], step instructions, step tests, step setup, step artifacts, per-step verifier results, min-reward gating, and mean/final reward strategies. The current DeepSWE repo has no steps/ task directories.
MetricsDefault metric is mean. Pier also supports sum, min, max, reward distributions, exception stats, token/cost totals, and pass@k when each task attempt has a single binary reward.
ArtifactsTrials land under jobs/<job>/<trial>/ with logs, agent trajectories, verifier output, artifacts, config, and result JSON.

DeepSWE/Pier scoring

The primary DeepSWE score is pass rate: a task trial passes when the verifier reward is 1. Pier’s generic reward model is a dictionary of numeric rewards; job-level metrics aggregate those dictionaries.

  • Trial score: VerifierResult(rewards={"reward": 1.0}) or {"reward": 0.0} for DeepSWE-style binary tasks.
  • Job score: mean reward by agent/model/dataset unless other metrics are configured.
  • Pass@k: computed when repeated attempts exist and each trial has exactly one binary reward.
  • Failure accounting: exceptions are tracked separately from rewards; missing verifier results count as non-success for pass@k.

Our harness-evals mechanics

Test setup

harness-evals is configured by harness-evals.yaml and YAML test cases, usually under evals/tests/**/*.yaml. A one-step case can use top-level prompt and assert; multi-step cases use ordered steps:.

id: checkout-refactor
suite: regressions
workspace:
  fixture: evals/fixtures/checkout
mocks:
  cli:
    jira-cli: jira-success
steps:
  - id: plan
    prompt: Review the checkout flow and propose a minimal plan.
    assert:
      - type: contains
        value: plan
  - id: implement
    prompt: Implement the approved change.
    assert:
      - type: exitCode
        equals: 0
      - type: workspaceDiff
        minChanged: 1

Run lifecycle

  1. Load config and cases. Defaults are merged with project config; one-step cases normalize to a generated run step.
  2. Build matrix. Selected cases are crossed with selected agents, with case and CLI overrides applied.
  3. Resolve Docker image. A ready image is probed, or a managed image is built from selected adapter install recipes.
  4. Copy workspace. The source repo or fixture is copied into a run-local workspace; the original repo is not edited.
  5. Execute steps sequentially. Each step runs the selected adapter command inside Docker. Later steps share the same copied workspace and receive adapter continuation.
  6. Parse events and mock calls. Adapters parse stdout/stderr into structured events; CLI/MCP mocks record matched and unmatched calls.
  7. Evaluate assertions. Built-ins run first; llmJudge assertions run afterward with configured or adapter-backed judge models.
  8. Score and report. Step and scenario scores are calculated, artifacts are written, and latest HTML/JSON/CSV reports are rendered.

Assertion and pass/fail semantics

Assertion typeWhat it checksBest use
exitCodeAgent process exit code.Agent CLI/runtime health, or command-adapter baselines.
contains / notContainsFinal step output text.Output regressions and expected status messages.
toolCalledParsed agent tool-call events.Workflow behavior, e.g. using read/edit tools.
mockCalledRecorded CLI/MCP mock calls.External integration contracts without touching real services.
noToolErrorsTool-call error flags.Detect bad tool usage.
workspaceDiffAdded/changed/deleted files relative to initial workspace snapshot.File-level edit boundaries and focused regressions.
settingsDrivenSetupPi settings setup and lack of explicit extension flags.Pi-specific setup policy.
llmJudgeRubric-based score over selected inputs.Quality, factuality, or implementation review that deterministic assertions cannot capture.

A step passes only when all required assertions pass. A run is passed only when all steps pass; failures, timeouts, and errors cause later steps to be skipped.

Scoring

harness-evals computes normalized weighted score buckets:

  • assertionPassRate: passed non-judge assertions divided by total non-judge assertions.
  • judgeScore: average score from llmJudge assertions.
  • latency: normalized duration, default minimize with worst 600,000 ms.
  • cost: normalized cost, default minimize with worst 1 USD.
  • tokenUsage: normalized token count, default minimize with worst 200,000 tokens.

Default weights include assertion pass rate and judge score with weight 1 each; latency, cost, and token usage are reported with weight 0 unless the project opts in.

Artifacts and reporting

The file output provider writes per-run directories under .harness-evals/runs/ and latest reports under .harness-evals/output/latest/. Per-step artifacts include stdout, stderr, command metadata, event summary, assertions, score, cost, mock calls, judge records, and step completion payloads.

Important gap relative to Pier: harness-evals does not currently have a first-class independent post-agent verifier phase that runs hidden project tests and emits a reward file. It can ask the agent to run tests, inspect output, assert workspace diffs, and use LLM judges; those are not equivalent to a separate deterministic verifier.

Reusable project-level harness for comparing real agent CLIs on workflows, regressions, and repo-specific tasks. Different center of gravity: benchmark validity vs project regression/observability.

Workspace setup Task image contains cloned upstream repo at base commit; task metadata pins exact commit. Copies project source or case fixture into a run-local workspace and mounts it into Docker. DeepSWE gives stronger external reproducibility. Our copied-fixture model is safer for local projects and leaves source untouched.
Agent runtime mini-swe-agent for leaderboard standardization; Pier also runs native CLI agents. Native CLI adapters are first-class: Pi, Claude Code, Codex, Cursor, command, and custom adapters. Our harness is more native-agent oriented; Pier’s standard harness choice reduces leaderboard confounds.
Internet and dependencies DeepSWE tasks disable task internet. Pier adds agent network allowlists so installed agents can still reach model endpoints. Controls forwarded environment variables. Docker network isolation/allowlisting is not a first-class scoring feature. Borrow Pier-style network policy for stronger isolation in high-stakes or benchmark-style cases.
Test setup Hidden test.patch adds tests and a test.sh; verifier script applies patch at grading time. Assertions are declared directly in case YAML; mocks and fixture files are also project-local. Our assertions are transparent and fast to author. Hidden test patches are better for black-box code behavior evaluation.
Verification independence Verifier runs after the agent, separate from the agent’s self-verification behavior. Assertions evaluate captured run state. The agent may run tests, but the harness itself does not run a separate project verifier by default. This is the biggest correctness gap if we want DeepSWE-like code-benchmark confidence.
Behavior vs implementation DeepSWE explicitly designs tests around observable behavior and public APIs. workspaceDiff can enforce exact files, which is useful for workflow regressions but can become implementation-shape-sensitive. Use exact diffs sparingly for correctness. Add behavior-verifier support for broader acceptance.
Regression protection Verifier runs selected baseline tests plus challenge tests; patch breaking unrelated behavior fails. Regression protection depends on authored assertions, mocks, and optional prompts asking the agent to run checks. Our framework should support a first-class verify command/assertion to run baseline and challenge tests after edits.
Scoring model Reward dictionary; default mean; binary pass rate and pass@k for repeated attempts. Weighted 0..1 score buckets across assertions, judges, latency, cost, and tokens. Our scoring is richer for tradeoffs. Add pass@k and reward-file compatibility for benchmark-style runs.
Multi-step evals Pier supports them; current DeepSWE corpus inspected is single-instruction tasks. Multi-step scenarios are core and share the same copied workspace with adapter continuation. Our multi-step support is a clear strength for real agent workflows.
Mocks and external systems Pier task environments can define MCP servers and network policies; DeepSWE focuses on repo-contained verifiers. Strict CLI and MCP mocks with recorded calls and mockCalled assertions. Our mock system is more directly useful for enterprise/project workflow evals.
Artifacts Trial result JSON, logs, trajectories, verifier output, artifacts such as model patch. Records JSONL, result/summary JSON, workspace diff, per-step stdout/stderr/events/assertions/score/cost, report files. Both are strong. Add patch-format model diff as a convenience artifact in our runner.
Trajectory analysis Augmented ATIF trajectory viewer and pier critique run with fresh-agent critiques. Reports show parsed events/tool calls; llmJudge is assertion-scoped during the run. Post-hoc critique mode would complement our built-in judges.
QA process Task authoring includes reference solution, flake checks, LLM-assisted analysis, and human review. Authoring guidance encourages goal interview, focused fixtures, deterministic assertions, mocks, and report triage. Adopt DeepSWE’s prompt-verifier bijection and flake-check checklist for high-quality cases.

Recommendations for harness-evals

1) Add a first-class post-agent verifier phase

Introduce a deterministic verifier that runs after the agent step against the edited workspace. It should support commands, timeouts, env, Docker user/workdir, artifacts, and reward.txt/reward.json parsing.

verify:
  command: bash evals/verifiers/checkout.sh
  timeoutMs: 300000
  rewardFile: .harness/reward.json
  artifacts:
    - coverage/
assert:
  - type: reward
    key: reward
    min: 1

2) Support hidden-test-patch style fixtures for benchmark cases

Add an optional task package format or verifier helper that can apply a project-local patch to a copied workspace after the agent finishes, then run base and challenge tests. This would make DeepSWE-style behavioral verification possible without abandoning YAML scenarios.

3) Add attempts and pass@k aggregation

Add --attempts or case-level attempts and calculate pass@k when pass/fail outcomes are binary. This aligns with benchmark practice and captures stochastic agent reliability better than one run per case.

4) Persist model patches as first-class artifacts

We already compute workspace diffs. Emit a patch file such as model.patch alongside workspace-diff.json for easier review, replay, and external judging.

5) Add Pier-style network isolation controls

Keep the existing environment variable allowlist, but add Docker network modes and per-agent egress allowlists for model/provider endpoints. This matters for tasks where accidental internet access can contaminate results.

6) Formalize a high-quality eval authoring checklist

Adopt DeepSWE’s QA vocabulary directly: prompt-verifier bijection, acceptance breadth, realism, environment cleanliness, flake checks, and reference solution review. This can live in skills/harness-evals/docs/writing-evals.md.

Bottom line: Keep harness-evals as a flexible project workflow harness, but add a Pier-inspired verifier/reward layer for code-behavior correctness. That combination would cover both local regression evals and benchmark-grade software engineering tasks.
